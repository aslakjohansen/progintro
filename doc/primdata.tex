Keeping track of a single value is trivial. But once you start having a handful of values to juggle, you begin to feel a burden. Yet, by following \idx{good practices}{Practice!Good} for naming your variables, it shouldn't be a big deal. However, as the number of values that we need to deal with grow, that solution model starts to break down. Imagine a program that we are writing a program that has to do some sort of analysis on the historical daily temperatures some calendar year. That is 365 (or 366) values right there, and that isn't even a remotely large number of values. Even with a good \idx{naming convention}{Convention!Naming}, keeping track of that many variables will be a mess. But it doesn't stop there: Some values are related and we often have to treat them that way. In this chapter we look into how we can structure data in ways that allows us to reason about much larger number of values through code. This is a form of \idx{scalability}{Scalability}.

\section{Complex Types}

% atomic types: types that cannot meaningfully be subdivided further, yes an int can be split into bits but why would I care if the 7th bit is set? it gets a bit more murky with floats, here I might want to look into the sign or the magnitude (exponent), but all the float instructions of our processor operate at the unit of a float, so atomic is an apt qualifier for such types (real atoms also consists of smaller elements), in \csharp\ they are called primitive
Primitive types cannot meaningfully be subdivided further. Yes, an \typename{int} can be split into bits, but why would I care what the value of the 7th bit is. It gets a bit more murky with floats. Here, I might want to look into the sign or the magnitude (i.e., the exponent). But all the floating-point instructions of our processor operate at the unit of a full floating-point value (i.e., sign, mantissa and exponent). So, atomic is an apt qualifier for such types. After all, real \idx{atoms}{Atom} also consists of smaller elements. In \csharp, such types are called \defi{primitive}{Type!Primitive}.

% complex types: complex types are are complexes of data, they contain (or at least may contain) multiple elements of data, these data can in turn be atomic or complex, those complex types can arrange and expose their elements in different ways, this chapter will cover the most basic variations, because these types structure data we call them datastructures
Complex types are complexes of data. They contain (or at least may contain) multiple elements of data that can in turn be atomic or complex. Those complex types can be used to arrange and expose these elements in different ways. This chapter will cover the most basic variations. Because these types structure data, we call them \defi{datastructures}{Datastructure}.

\section{Reference Types}

% problem: from the compilers point of view atomic types are easy to work with, an atomic type fits into a register and can be read from and written to memory through a single operation, a complex type on the other hand does not fit into a single register, and so the compiler has to decide how large a part of the data needs to be loaded from memory to registers how and how much needs to be written back after we are done with it, later on you will learn that multiple things can go on at the same time within the same program, so the implementation of this needs to ensure that a consistent view of the data is presented to all these ongoing things
From the \idx{compilers}{Compiler} point of view, atomic types are easy to work with. They fit into a \idx{register}{Register}, and can be read from and written to \idx{memory}{Memory} in a single operation. Choosing one of two values is trivial. A complex type, on the other hand, does not fit into a single register, and so the compiler will have to decide how large a part of the data needs to be loaded from memory into registers, and how much needs to be written back after we are done with it. Later on, you will learn that multiple things can go on at the same time within a single program. When this happens, the implementation of this needs to ensure that a consistent view of the data is presented to all of these ongoing things.

% solution (ref type): references do fit into a single register, variables of reference types holds the reference (address) and not the value, two variables may refer to the same data (by holding the same reference), nothing changes with respect to the data being referenced, it still doesn't fit into a single register, but copying the reference is a trivial and fast operation
A reference is an \idx{address}{Address!Memory} (in memory) to some datastructure, and such a reference can be used as a value. References are atomic values and do fit into a single register. Variables of reference types thus hold the reference (i.e., the address) and not the value (i.e., the datastructure). Two reference type variables can refer to the same data. All it takes is for them to hold the same address. Nothing changes with respect to the data being referenced: It still doesn't fit into a single register. But we now have the option of copying references, and that operation is both trivial and fast.

% solution (acceptance): another solution is simply to accept that the datastructure cannot be represented in a single register and deal with the complications that stem/arise from that, this is essentially what you have done so far if you have had two related variables and you needed to make a copy of the combination, luckily those complications are usually handled by the compiler, \csharp choices
Another solution is simply to accept that datastructures cannot be represented in a single register, and deal with the complications that arise from that. This is essentially what you have done so far when you had two variables and needed to make a copy of that combination. Luckily, those complications can be handled by the compiler. So, while making datastructures reference types is the natural choice, they can also be made value types. \csharp\ offers mostly reference type datastructures, and these are the ones we will be focusing on.

% summary
So, types where a variable is mapped to hold values in register(s) are called \idx{value types}{Value type}, and types where variable is mapped to hold a reference to an address in memory is called a \idx{reference type}{Reference type}. Copying a reference type copies the reference, and any modification to whatever it refers to is visible through all reference copies to these data.

Let's start with an example. We need to create a program that helps us navigate a house. For this we need to model the concept of a door. A door, in the context of the this problem, has a height of type \typename{int}, a width of type \typename{int} and \typename{bool} for wether it is open. We would like to be able to represent multiple doors and to refer to them individually. This example of a door is an example of \idx{structured data}{Structured data}, or -- as they are often called -- structs. To generalize, \defi{structs}{Struct} are groupings of typed and individually named values that are stored contiguously in memory.

\subsection{Colors}

% light as a spectrum
\defix{Light} is made up of photons, and each \idx{photon}{Photon} has a wavelength. The \defi{wavelength}{Wavelength} of the photons determines which color our eyes will observe. A wavelength of 380nm is \idx{violet}{Violet} and one of 740nm is \idx{red}{Red}. In between these you have all the colors of the \idx{rainbow}{Rainbow}, in the order of the rainbow. Coincidence? Just outside of the \idx{visible spectrum}{Spectrum!Visible}, you will find \idx{infra-red}{Infra-red} and \idx{ultra-violet}{Ultra-violet}. Our eyes are, however, constantly bombarded by photons, so in reality we always observe a distribution across this spectrum. Our \idx{eyes}{Eye}, though only has three (or four) tunings of receptors. These, roughly corresponds to red, green and blue. So, your eyes (and \idx{brains}{Brain}) need to do a lot of interpretation.

% computer representation
In a computer, colors are typically \idx{represented}{Representation!Color} using three values; one for red intensity, one for green intensity, and one for blue intensity. That somehow matches the model of our eyes \ldots\ We say that these are the \textsl{components} of the color. There is a number of ways to express such an \idx{intensity}{Intensity}. It seem natural to use a number between 0 and 100, or perhaps a number between 0.0 and 1.0. However, due to the way processors work on integers, these boundaries are not particularly special. Typically, a byte is use to represent each component of the color. That means that we have the ability to express $2^8=256$ different intensities of each component, or a total of $256^3=16777216$ different colors. The value zero is used to represent complete \idx{darkness}{Darkness}. Full intensity is then the value 255.

% color spaces
This constitutes a \defi{color space}{Color!Space} whereby all colors fit within a \idx{color cube}{Color!Cube} spanned by the red, green and blue color vectors. Hence, it is named the \idxx{RGB} color space. Figure \ref{fig:primdata:struct:color} depicts this space. It is a direct fit for how \idx{GPUs}{GPU} represent \idx{images}{Image}. Other color spaces exist though, and there are good reasons for their existence. Other popular color spaces include \idxx{HSV} (which is shaped like a cylinder) and \idxx{XYZ} (which is shaped like an tongue). Formulas exists for \idx{converting}{Color!Space!Converting} between these. Each of these color spaces represents practical simplifications of what light really is.

Most functional languages make heavy use of \idx{pattern matching}{Pattern matching}. This makes the structure of linked lists become very apparent.
