\subsection{Blocks}

% problem: technically branching on a single statement is all we need, but it is not particularly convenient, imagine us having to do two things if a condition holds true: we will have to check a condition and if true do the first thing, then we will have to check the same condition again and if true do the second thing, this comes with three significant downsides, namely: (i) the condition needs to be evaluated twice and this takes time and could involve complex side effects, (ii) the code would quickly get complex to understand, and (iii) it would get significantly more convoluted if the first thing modifies a variable that takes part in the condition
Technically, branching on a single statement is all we need. But this would not be particularly convenient. Imagine us having to do two things if a condition holds true: We would have to check a condition and if true do the first thing. Then we would would have to check the same condition again, and if true do the second thing. This comes with three significant downsides, namely:
\begin{enumerate}
  \item The condition needs to be evaluated twice. This takes some amount of time and could involve complex side effects.
  \item The code would quickly get complex to understand.
  \item It would get significantly more convoluted if the first thing modifies a variable that takes part in the condition.
\end{enumerate}

% solution: most imperative programming language have the notion of a block, this is a sequence of statements that take the place of a single statement, so by employing a block the one statement at the true branch of an if statement could become a sequence of statement, and likewise of the false branch
Most imperative programming languages have a notion of a \defi{block}{Block}. This is a sequence of statements that take the place of a single statement. So, by employing a block the one statement at a true branch of an \keywordname{if} statement could become a sequence of statements. And likewise for every syntactic occurrence of a statement (e.g., the false branch of an \keywordname{if} statement).

\subsection{Chaining Branches}
\label{sec:flow:branch:chaining}

% common pattern: define pattern (instead of choosing one of two paths we may want to chose one of a larger number of paths), we can implement this as a sequence of cascading if statements where the conditions are evaluated one after the other until one matches, then the statement associated with that condition is evaluated
The \keywordname{if} construct allowed us to chose one of two paths. But sometimes we might want to choose one of three (or more) paths. We can achieve this by \idx{nesting}{Nesting} \keywordname{if} constructs. The trivial way of implementing this is to create a sequence of \idx{cascading}{if statement@\keywordname{if} statement!Cascade} \keywordname{if} statements where the conditions are evaluated one after the other until one matches (or the final \keywordname{else} branch is reached). Then the statement associated with that condition is matched.

\subsection{\keywordname{switch} and \keywordname{case}}

% evaluation of chained branches as a tool
Let's revisit how those \idx{chained branches}{Branch!Chain of} are evaluated. One condition is evaluated at a time. As this takes time, we would ideally like to avoid it. This particular example employs a common pattern: Each of the chained conditions compare the equality of the same variable. It doesn't take much to imagine a different (higher \idx{level}{Abstraction!Level}) construct that limits this redundancy and allows you to express the same logic in less code. Such a construct would have the added benefit of directly telling the programmer what it is used for. In the previous example, as a contrast, you had to read lines 3, 5, 7 and 9 in order to understand what the purpose of these nested construct were.

% the switch alternative
So, what would such a construct look like? What kinds of elements does it need? First of all, we need to know which variable we are comparing. But since we only need to know about it once, it makes sense to \idx{generalize}{Generalization} this to being an \idx{expression}{Expression}. Then, we need a list of \textsl{cases}; values to compare against and logic that needs to evaluated should we have a match. The values can be generalized to expressions and the logic to a \idx{statement}{Statement}. On top of that we need some syntax to allow the \idx{compiler}{Compiler} to \idx{parse}{Parsing} it.

\subsubsection{\csharp}
\label{sec:flow:branch:switch:cs}

What does this look like for \csharp? The first rule of syntax \ref{syntax:flow:branch:switch} looks fairly complicated. It does get a lot less scary once you realize two things though: Firstly, there is a loop which for one of more cases mapping a concrete value (through an expression) to a statement. Secondly, after this loop there is a \idx{\textsl{default}}{Default!Case} statement that can be bypassed. That is, we \defi{\textsl{switch}}{switch statement@\keywordname{switch} statement} on an expression, then we have a number of cases, these are potentially followed by a single default case and finally the \keywordname{switch} block is ended.

% rules: break
The second syntax rules introduces a \keywordname{break} statement. By default, the evaluation of the matching case statement \idx{falls through}{Fall through} to the next case statement (which is then evaluates). This continues all the way through the entire switch construct. That is, the matching case statement and all following statements are evaluated in order. By considering the order, this allows one to construct some very attractive behaviors. However, in most cases this is confusing behavior, and we thus want to make sure that the evaluation doesn't fall through. We do this by inserting a \idx{\keywordname{break} statement}{break statement@\keywordname{break} statement}. Additionally, \csharp\ has a hard rule that the last ending \idx{curly brace}{Brace!curly} of the \keywordname{switch} construct must not be reachable. For now, that means adding a \keywordname{break} statement to the end of it. Later on, in section \ref{sec:exceptions}, we will introduce another \idx{exceptions}{Exception}, which is another way to get around this rule.
