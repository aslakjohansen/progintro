\subsection{\keywordname{if} and \keywordname{else}}

% intro: the structure of a conditional, its a fork in the road, you are either in the situation where the condition evaluates to true (and you go down one road) or you are in the situation where the condition evaluates to false (and you go down a different road), eventually these roads will merge, for now they will merge after the evaluation of a single statement
As illustrated in figure \ref{fig:flow:branch:generic}, the decision on which branch to take is made based on the evaluation of a condition. That condition evaluates -- by definition -- to either \valuename{true} or \valuename{false}. As there is two possible outcomes, this represents two possible futures; a fork in the road. You are either in the situation where the condition evaluates to \valuename{true} (and you go down one road), or you are in the situation where the condition evaluates to \valuename{false} (and you go down a different road). Eventually, these roads will merge. This happens after the evaluate of a single statement.

\subsection{Blocks}

% problem: technically branching on a single statement is all we need, but it is not particularly convenient, imagine us having to do two things if a condition holds true: we will have to check a condition and if true do the first thing, then we will have to check the same condition again and if true do the second thing, this comes with three significant downsides, namely: (i) the condition needs to be evaluated twice and this takes time and could involve complex side effects, (ii) the code would quickly get complex to understand, and (iii) it would get significantly more convoluted if the first thing modifies a variable that takes part in the condition
Technically, branching on a single statement is all we need. But this would not be particularly convenient. Imagine us having to do two things if a condition holds true: We would have to check a condition and if true do the first thing. Then we would would have to check the same condition again, and if true do the second thing. This comes with three significant downsides, namely:
\begin{enumerate}
  \item The condition needs to be evaluated twice. This takes some amount of time and could involve complex side effects.
  \item The code would quickly get complex to understand.
  \item It would get significantly more convoluted if the first thing modifies a variable that takes part in the condition.
\end{enumerate}

% solution: most imperative programming language have the notion of a block, this is a sequence of statements that take the place of a single statement, so by employing a block the one statement at the true branch of an if statement could become a sequence of statement, and likewise of the false branch
Most imperative programming languages have a notion of a \defi{block}{Block}. This is a sequence of statements that take the place of a single statement. So, by employing a block the one statement at a true branch of an \keywordname{if} statement could become a sequence of statements. And likewise for every syntactic occurrence of a statement (e.g., the false branch of an \keywordname{if} statement).
