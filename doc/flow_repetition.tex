\section{Repetition}
\label{sec:flow:repetition}

% problem: print something once, twice, 1000 times, scalability problem, variable number of times
Let's say that we want to print out \say{Hello} once. This is a single trivial line of code. If we want to do it twice, then we can duplicate that line. That solution will still technically work if we want to print it out 1000 times, but then we are left with a 1000 line program that is hard to understand (lots of \idx{scrolling}{Scrolling} is necessary) and, frankly, quite boring. This is a problem of \idx{scalability}{Scalability}. But it gets worse: Imagine that we have an integer variable that has been assigned a value from a user input, and we want to print out \say{Hello} this many times. As the concrete value isn't known at \idx{compile-time}{Compile time}, the strategy of copying the line and pasting the right number of times simply cannot work.

% solution
To be able to handle this type of problems, programming languages provide a new class of constructs. These implement repetition of logic, or \idx{loops}{Loop}. \csharp\ provides three main forms of loops. Syntax \ref{syntax:flow:repetition} illustrates what these look like in code, and we will be spending the next couple of sections going through each of them. In section \ref{sec:primdata:sequence:foreach}, we will go through a fourth form that works at a higher \idx{level of abstraction}{Abstraction!Level}.

\subsection{\keywordname{while}}

% highlevel concept: the while loop is the most fundamental loop, speaking aloud explanation (as long as a condition is true)
The \idx{\keywordname{while} loop}{while loop@\keywordname{while} loop} is the most fundamental loop; it is the first you learn as an \idx{assembly}{Assembly programming languages} programmer, and most simple to construct in assembly language. In a \keywordname{while} loop, something is done \textsl{while} a condition holds.

\subsection{\keywordname{do-while}}

% highlevel concept: speaking aloud explanation, notice that the statement comes before the condition, use this as a mnemonic device for the order of evaluation: the statement is evaluated first, this gives us an at-least-once property
The \keywordname{while} loop checked a condition and evaluated a statement. But what if we need to evaluate the statement first? We could place a copy of the statement before the loop, but -- as we have already covered -- \idx{duplicated code}{Duplicate code} comes with its own share of problems. Some languages, like \csharp, deals with this by including a so-called \idx{\keywordname{do-while} loop}{do-while loop@\keywordname{do-while} loop}. The \idx{semantics}{Semantics} are exactly the same as what we have seen in the \keywordname{while} except that the evaluation order is flipped: First the statement is evaluated, and then the condition is checked.

% figure: flowchart

\subsubsection{\csharp}

% syntax
In \csharp, \keywordname{do-while} loops follow the second rule of syntax \ref{syntax:flow:repetition}. Notice how the statement comes before the condition. That is not by accident. This was an intentional design decision to make it more natural for humans to read. The \idx{compiler}{Compiler} (or those writing it) wouldn't care. But let's see what using it looks like in practice \ldots

% setup
Consider a problem where we have an integer number in one variable, and we need to print out all the individual digits in order of growing significance. One reasonable solution to to express this using a \keywordname{do-while} loop. That looks like this:

% example
\inputminted[linenos]{csharp}{../src/csharp/flow_repetition/dowhile/Program.cs}

% explain
The first line declares a variable and initializes it to a value. This is the value that we want to print out one \idx{digit}{Digit} at a time. Then comes the \keywordname{do-while} loop. Here, the \idx{mnemonic}{Mnemonic} is that as the loop body comes before the condition in the code, it will also be \textsl{evaluated} before the condition. So, the loop body will be evaluated at least once, and it will stop evaluating once \varname{i} reaches zero. For each \idx{iteration}{Iteration}, line 4 divides \varname{i} by ten (i.e., removed the least significant digit) and updates \varname{i} to reflect this new value. Accordingly, the value of \varname{i} will decrease for each iteration and thus eventually reach zero (aka the loop \idx{exit condition}{Exit condition!Loop}). Line 3 -- which is evaluated before \varname{i} is updated -- takes \idx{modulo}{Module operator} 10 of \varname{i} (i.e., the least significant digit) and prints this to the screen. So, the loop body essentially chunks off the least significant digit and prints it out.

\subsubsection{Problem of Repetition}

% motivation: so now we have two ways of doing the same thing, that begs the question: which option to go for? more importantly why should one option be chosen over the other? which qualities should we use for "scoring" each of the two options?
So, now we have two ways of doing (more or less) the same thing. That begs the question: Which option should we pick? More importantly, why should one option be chosen over the other? And which qualities should we use for \textsl{scoring} each of the two options?

% problem: two qualities stick out: readability and maintainability, readability essentially means how easy the code is to read and maintainability on the other hand boils down to how easy the code is to maintain, we can employ a number of (more or less well-defined) metrics for each, in this particular situation they both point in the same direction, this stems for one option having a duplicate element, for instance the way we went from a do-while to a while was to copy out a copy of the loop body before the loop, from a readability perspective this creates more lines burdens us with the need to recognize that those two pieces of code are identical and part of the same loop (this is explicit in the other option), from a maintainability perspective duplicated code is always risky as the developer is burdened with the need to manually keep the two copies in sync, any failure to do so will result in a bug, in summary unless you have a good reason not to you should simply use the most compact version
Two qualities stick out: Readability and maintainability. \defi{Readability}{Readability} essentially means how easy the code is to read, and \defi{maintainability}{Maintainability} boils down to how easy the code is to maintain. We can employ a number of (more or less well-defined) \idx{metrics}{Metric} for each. In this particular situation, they both happen to point in the same direction. This stems from one of the options having a duplicate element. For instance, the way we went from a do-while loop to a while loop was to place a copy of the loop body before the loop. From a readability perspective this creates mode lines of code, and it burdens us with the need to both recognize that those two pieces of code are identical and that they are part of the same loop. The latter part is explicit in the do-while option. From a maintainability perspective, duplicated code is always \idx{risky}{Risk} as the developer is burdened with the need to manually keep the two copies in \idx{sync}{Sync}. Any failure to do so will result in a bug. In summary, unless you have a good reason not to, you should simply use the most compact version.

\subsection{\keywordname{for}}

% common use case: it didn't take long for a common pattern of while loop usage to emerge, in this pattern a loop variable is declared before the loop, at the end of the loop this variable is adjusted and the condition checks whether this variable has passed some value
It didn't take long for a common \idx{pattern}{Pattern} of while loop usage to emerge. In this pattern, a loop variable is declared before the loop, at the end of the loop this variable is adjusted and the condition check whether this variable has passed some value. This is, for instance, how you would loop through all integers between two values.

% defining template: given this pattern a template can ben synthesized: initialization statement, condition, update statement, body, by generalizing the template we ensure that it is as applicable as possible, a new kind of loop -- the for loop -- is designed specifically to accommodate this template
If generalized, a template can be synthesized from this pattern. We have an \idx{initialization statement}{Initialization statement}, a \idx{continuation condition}{Condition!Continuation}, an \idx{update statement}{Update statement}, and a \idx{loop body}{Loop!Body}. The generalization comes from us using the generic construct of a statement for initialization and updates. Note that by choosing to build on statements, we also gain the option of having the initialization and update statements be expressed without any shared variables. By generalizing the template we ensure that it is as \idx{applicable}{Applicability} as possible. A new kind of loop -- the \keywordname{for} loop -- is designed specifically to accommodate this template.

\subsection{Control Structures}

% problem: loops are control structures that repeat, in principle we can define the continuation condition on arbitrary complexes of logic, but that makes the code difficult to read and is rarely done, instead loops are typically used to draw the broad strokes: "we need something that repeats here", once that control logic has been set up it will churn trough each repetition (aka iteration) and we can focus on what happens *during* these iterations, on top of this we have two ways of deviating from this pattern.
Loops are control structures that repeat statements. In principle, we can define the continuation condition on arbitrary complexes of logic, but that makes the code difficult to read and is rarely done. Instead, loops are typically used to draw the broad strokes: \say{We need something that repeats here}. Once that control logic has been set up, it will churn through each repetition (aka \idx{iteration}{Iteration}) and we can focus on what happens \textsl{during} these iterations. On top of this, we have to two ways of deviating from this \idx{pattern}{Pattern}.

% break: the first such method of deviation is to *break* the loop. At any point inside the loop we can decide (often using an if statement) to end the loop prematurely, the evaluation of the loop will stop and the remainder of the loop will be skipped, this is useful if you are searching for something, once we have found it we break the loop.
The first such method of deviation is to \textsl{break} the loop. At any point inside the loop, we can decide -- often using an \keywordname{if} statement -- to end the loop prematurely: The \idx{evaluation}{Evaluation!Breaking loop} of the loop will stop and the remainder of the loop (rest of current and all future iterations) will be skipped. This is useful if we are \idx{searching}{Searching} for something: One we have found it, we break the loop.

% continue: the second such method of deviation is to skip the remainder of the current iteration and *continue* at the beginning of the next, this is often used to skip special cases.
The second method of deviation is to skip the remainder of the current iteration and \textsl{continue} at the beginning of the next. This is often used to skip special cases.

\subsection{Nested Loops}

% intro
Loops can be used in many different ways, but unless you start stretching definitions then what they do boils down to iterating over a single dimension; a single sequence of values. But sometimes that is not enough. The \idx{pixels}{Pixel} of a \idx{screen}{Screen} are laid out in a 2d \idx{grid}{Grid}. If we want to perform some operation on all pixels, then we need to do through two dimensions. In this particular case, those two dimensions are often \idx{serialized}{Serialization} to one dimension. In other situations that is not the case. Take, for instance, a multiplication table. Here, we want to do something with all combinations for two sets of \idx{sorted}{Sorting} integers. First, we want to iterate over all elements from the first set, and, for each of these, we want to iterate over all the elements from the second set. We do this by placing one loop inside the body of another. This is called \defi{nesting}{Loop!Nesting} one loop inside another.
